# ic-memory 0.6 Design: Long-Term Stable-Memory Protocol

## Status

Draft for 0.6.

## Goal

`ic-memory` 0.6 turns the crate from a validated allocation ledger into a long-term stable-memory protocol.

The priority is not backward compatibility. Nobody depends on the current durable format yet. The priority is to establish the format, authority model, evolution discipline, and testing strategy that can survive years of upgrades, rollbacks, schema evolution, and framework integrations.

The central invariant remains:

```text
stable_key -> allocation_slot forever
```

0.6 extends that with a temporal invariant:

```text
durable bytes must never be interpreted under the wrong semantic version
```

## Motivation

The 0.5 audit found that the default path is now materially stronger: allocation authority is validation-bound, corrupt stable-cell bytes are classified, unsupported versions fail closed, and Canic delegates bootstrap/opening through `ic-memory`. The remaining high-risk area is future evolution discipline: custom compatibility readers and derived serde DTOs can make it easy for future versions to accidentally accept newer durable formats through older semantics.

0.6 should therefore remove ad hoc compatibility ranges and replace them with
crate-owned recovery provenance.

## Design Principles

### 1. Stable memory is a protocol, not a struct dump

The durable format is protocol surface.

This includes:

```text
ledger payload envelope
physical commit store
stable-cell ledger record
allocation ledger
allocation slot descriptor
stable-key grammar
range authority ordering
macro registration semantics
validated allocation authority
```

Changing any of those requires explicit versioning and recovery/evolution logic.

### 2. Decode must not happen before version routing

0.5 currently has the classic risk:

```text
deserialize -> inspect version
```

0.6 must invert that:

```text
select committed generation
-> read ledger payload envelope
-> route to versioned reader
-> recover/validate
-> authority
```

### 3. Compatibility is not caller-supplied

Remove or deprecate arbitrary APIs like:

```rust
recover_with_compatibility(...)
commit_with_compatibility(...)
```

Replace them with crate-owned recovery entry points. Add named migration paths
later only when a second durable schema exists.

No downstream caller should be able to widen compatibility by passing a permissive range.

### 4. Authority must have provenance

`ValidatedAllocations` should only be produced from:

```text
physical committed generation
-> logical payload envelope
-> schema-specific reader
-> recovered ledger proof
-> committed-integrity validation
-> declaration snapshot validation
-> policy validation
-> authority capability
```

The type should remain opaque, non-serializable, and impossible to construct outside the crate.

# Proposed Architecture

## 1. Ledger Payload Envelope

Introduce a logical ledger payload envelope that is intentionally small and
manually decoded before any allocation-ledger DTO.

This envelope wraps the logical payload inside each physical committed
generation. It does **not** replace the stable-cell envelope, the
`StableCellLedgerRecord`, or the dual-slot physical recovery container.

The layering is:

```text
MemoryManager ID 0
-> ic-stable-structures Cell envelope
-> StableCellLedgerRecord
-> LedgerCommitStore / DualCommitStore
-> CommittedGenerationBytes
-> LedgerPayloadEnvelope
-> version-routed AllocationLedger
-> CanonicalLedger / RecoveredLedger
-> ValidatedAllocations
-> AllocationSession
```

```rust
pub struct LedgerPayloadEnvelopeHeader {
    pub magic: [u8; 8],
    pub envelope_version: u16,
    pub ledger_schema_version: u32,
    pub physical_format_id: u32,
    pub payload_len: u64,
}
```

Suggested magic:

```text
ICMEMLED
```

The envelope allows 0.6+ code to classify logical ledger bytes before full
ledger decode:

```text
empty -> initialize genesis
bad magic -> corrupt / foreign memory
unsupported envelope version -> unsupported
unsupported ledger schema -> unsupported / future migration required
unsupported physical format -> fail closed
valid routed payload -> decode through versioned reader
```

The 0.6 envelope deliberately does not include a custom payload-encoding or
custom substrate hook. The logical payload is the current `ic-memory`
`AllocationLedger` CBOR format, and the allocation substrate is
`ic-stable-structures::MemoryManager` with `u8` IDs `0..=254`.

Physical recovery remains separate:

```text
stable-cell preflight
-> decode StableCellLedgerRecord
-> select highest valid committed generation
-> decode LedgerPayloadEnvelope from selected payload
-> route logical ledger by schema version
```

The existing physical checksum remains the torn-write and accidental-corruption
detector for the committed payload. A cryptographic payload hash can be added
later only if a concrete need appears.

## 2. Versioned Ledger Readers

Replace raw compatibility ranges with crate-owned versioned readers.

```rust
pub trait LedgerReader {
    const LEDGER_SCHEMA_VERSION: u32;

    fn decode(payload: &[u8]) -> Result<DecodedLedger, LedgerDecodeError>;

    fn validate(decoded: &DecodedLedger) -> Result<(), LedgerIntegrityError>;

    fn recover(decoded: DecodedLedger) -> Result<RecoveredLedger, RecoveryError>;
}
```

0.6 ships with:

```rust
CurrentLedgerReader
```

Recovery becomes:

```text
read physical slots
-> select highest valid committed generation
-> decode ledger payload envelope
-> route by ledger_schema_version
-> decode with exact reader
-> recover canonical runtime model
-> validate committed integrity
-> return recovered ledger authority
```

No caller can say “accept versions 1 through 9.”

## 3. Canonical Runtime Ledger

Separate durable DTOs from runtime authority state.

```rust
pub struct CanonicalLedger {
    records: Vec<CanonicalAllocationRecord>,
    generation: LedgerGeneration,
    schema_version_read_from: u32,
}
```

The runtime uses only canonical types after recovery.

Durable structs should not be the same structs used for authority decisions.

This avoids the trap where serde shape and runtime semantics become accidentally coupled.

## 4. Recovered Ledger Proof

`RecoveredLedger` is the proof object that a logical ledger payload has crossed
the recovery boundary.

```rust
pub struct RecoveredLedger {
    canonical: CanonicalLedger,
    physical_generation: LedgerGeneration,
    ledger_schema_version: u32,
    envelope_version: u16,
}
```

Only crate-owned recovery code can construct it.

`validate_allocations()` should accept this proof type, not raw
`AllocationLedger`:

```rust
pub fn validate_allocations(
    recovered: &RecoveredLedger,
    snapshot: &DeclarationSnapshot,
    policy: &impl AllocationPolicy,
) -> Result<ValidatedAllocations, ValidationError>;
```

That makes this boundary explicit:

```text
raw bytes
-> physical recovery
-> logical envelope decode
-> schema-specific reader
-> recovered ledger proof
-> validated allocations
```

## 5. Wire Schema Fixtures

0.6 must add golden fixtures before release.

Required fixtures:

```text
fixtures/v1/empty_genesis_payload_envelope.hex
fixtures/v1/single_active_allocation_payload_envelope.hex
fixtures/v1/reserved_allocation_payload_envelope.hex
fixtures/v1/retired_allocation_payload_envelope.hex
fixtures/v1/ledger_commit_store_single_active.cbor.hex
fixtures/v1/dual_slot_store_valid_newer.cbor.hex
fixtures/v1/dual_slot_store_corrupt_newer.cbor.hex
fixtures/v1/stable_cell_record.cbor.hex
fixtures/v1/memory_manager_descriptor.cbor.hex
```

Tests must prove:

```text
known fixture decodes
known fixture validates
known fixture recovers into RecoveredLedger
re-encoded current output is intentional
unknown fields do not silently mint authority
renamed fields break tests
enum shape changes break tests
unsupported versions fail closed
```

## 6. Stable-Key Grammar Freeze

For schema v1/v2, define the stable-key grammar as protocol syntax.

Example rule:

```text
[a-z][a-z0-9_]*(\.[a-z][a-z0-9_]*)*
```

Document:

```text
Changing this grammar is a ledger-schema change.
```

Current grammar validation must be version-aware:

```rust
StableKeyGrammar::for_schema(version).validate(key)
```

Do not validate historical keys using whatever the current source code happens to consider valid.

## 7. Descriptor Semantics

`AllocationSlotDescriptor` should become explicitly versioned.

```rust
pub enum AllocationSlotDescriptor {
    MemoryManagerV1 {
        id: MemoryId,
    },
}
```

For 0.6, `ic-memory` is specifically a `MemoryManager` ID registry wrapper for
`ic-stable-structures`. Allocation slots are `u8` `MemoryManager` IDs, with ID
255 reserved by `ic-stable-structures` as the unallocated-bucket sentinel.

```text
slot kind is MemoryManager only
descriptor version changes when MemoryManager descriptor interpretation changes
unknown descriptor shape fails closed during durable decode or validation
```

No custom substrate descriptor should be part of the 0.6 authority model.

## 8. Remove Arbitrary Compatibility APIs

Remove:

```rust
recover_with_compatibility
commit_with_compatibility
LedgerCompatibility { min, max }
```

Replace with:

```rust
recover_current(...)
commit_current(...)
```

If a testing API is needed, hide it behind:

```rust
#[cfg(test)]
```

or a feature:

```toml
dangerous-test-compatibility
```

Do not expose it as a normal advanced API.

## 9. Recovery Error Taxonomy

0.6 should distinguish:

```rust
pub enum RecoveryError {
    EmptyStore,
    ForeignStore,
    CorruptPayloadEnvelope,
    UnsupportedEnvelopeVersion,
    UnsupportedLedgerSchemaVersion,
    UnsupportedPhysicalFormatVersion,
    DecodeFailed,
    IntegrityRejected,
    PolicyRejected,
}
```

This matters because operators need to know whether the state is:

```text
new install
corrupt
too new
too old
foreign
upgrade/migration-needed
valid but policy-invalid
```

## 10. Canic Contract

Canic should depend on 0.6 using an exact version for release branches:

```toml
ic-memory = "=0.6.0"
```

Canic should treat these as protocol-sensitive:

```text
macro registration timing
range authority ordering
stable-key grammar
MemoryManager ID semantics
diagnostic projection
bootstrap order
```

Canic should not have an alternate authority path for opening memory.

Any legacy diagnostic path must be documented as read-only and non-authoritative.

# 0.6 Public API Shape

Preferred high-level API:

```rust
pub fn bootstrap_default_memory_manager_with_policy(
    policy: impl AllocationPolicy,
) -> Result<BootstrappedMemoryManager, RuntimeBootstrapError>;
```

Low-level API:

```rust
pub fn recover_current(
    store: &impl CommitStore,
) -> Result<RecoveredLedger, RecoveryError>;

pub fn validate_allocations(
    recovered: &RecoveredLedger,
    snapshot: &DeclarationSnapshot,
    policy: &impl AllocationPolicy,
) -> Result<ValidatedAllocations, ValidationError>;
```

Important change:

```text
validate_allocations() should not accept raw AllocationLedger.
```

It should accept only:

```rust
RecoveredLedger
```

or another crate-owned proof type that already means:

```text
version-routed
decoded
committed-integrity-checked
```

That seals the boundary.

# Release Criteria

0.6 is ready when all are true:

```text
No raw ledger can mint ValidatedAllocations.
No caller-supplied compatibility range exists in public API.
Envelope version is read before full ledger decode.
Golden wire fixtures exist.
Stable-key grammar is documented as frozen protocol syntax.
Slot descriptor semantics are MemoryManager-only and documented.
Unsupported future versions fail closed.
Rollback to older binaries cannot reinterpret newer state.
Canic has no alternate memory-opening authority path.
```

# Recommended Implementation Order

## 0.6 MVP

Keep the first 0.6 implementation intentionally small:

1. Add `RecoveredLedger` proof type.
2. Change `validate_allocations()` to accept `RecoveredLedger`, not raw
   `AllocationLedger`.
3. Remove caller-supplied compatibility ranges from the public API.
4. Add logical `LedgerPayloadEnvelope` inside `CommittedGenerationBytes`
   payloads.
5. Add one current 0.6 ledger reader.
6. Add golden fixtures for the 0.6 durable wire format.

## Deferred Work

Defer these until there is a concrete second schema need:

- migration registry;
- multi-schema migration graph;
- cryptographic payload hash;
- migration from 0.5 durable bytes, unless Canic has deployed state that needs
  it.

## Detailed Order

1. Add `RecoveredLedger` and make it crate-constructed only.
2. Route `validate_allocations()` through `RecoveredLedger`.
3. Remove public caller-supplied compatibility ranges.
4. Introduce `LedgerPayloadEnvelope` under `CommittedGenerationBytes`.
5. Add the current 0.6 ledger reader.
6. Add golden fixtures.
7. Freeze and document stable-key grammar.
8. Tighten Canic integration.
9. Run temporal compatibility audit again.

---

# Summary

0.6 should not be a feature release.

It should be the protocol-stability release.

The key change is this:

```text
0.5:
caller may help decide compatibility

0.6:
ic-memory owns compatibility and authority provenance
```

The final provenance chain should be:

```text
physical committed generation
-> logical payload envelope
-> schema-specific reader
-> recovered ledger proof
-> validated allocations
-> openable memory session
```

That is the correct long-term architecture.
